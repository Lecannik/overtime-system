# 🛡️ Отчет по анализу безопасности и уязвимостей проекта OvertimePro

На основе детального статического анализа исходного кода бэкенда (FastAPI) и фронтенда (React), а также результатов предыдущих аудитов безопасности, подготовлен перечень критических, высоких и средних уязвимостей, а также слабых мест архитектуры проекта.

---

## 🔴 P1 — Высокий приоритет (Критические риски безопасности)

### 1. Хранение Access JWT в `localStorage` (XSS-вектор)
* **Проблема:** Фронтенд сохраняет JWT access-токены в `localStorage` браузере (`frontend/src/services/api.ts`). Данные из `localStorage` доступны для чтения любому JavaScript-коду, выполняющемуся на странице. В случае успешной XSS-атаки (например, через уязвимости в сторонних npm-пакетах или некорректное экранирование пользовательского ввода), злоумышленник сможет мгновенно похитить сессию пользователя.
* **Рекомендация:**
  1. Перенести хранение `access_token` исключительно в оперативную память (in-memory state) React-приложения.
  2. Настроить механизм автоматического обновления токена по таймеру через `refresh_token`, который безопасно хранится в `HttpOnly` `Secure` cookie.

### 2. Неэффективный кэш токенов MS Graph (Фактически не работает)
* **Проблема:** В модуле `backend/app/services/ms_graph.py` класс `ConfidentialClientApplication` инициализируется при каждом входящем запросе к API. Соответственно, встроенный кэш токенов MSAL обнуляется при каждом обращении. Вызовы `acquire_token_silent` всегда возвращают `None`, вынуждая сервис при каждом запросе осуществлять полноценный сетевой обмен с серверами Microsoft за новым токеном. Это существенно замедляет отправку уведомлений/OTP и несет риски блокировки по rate-limit со стороны Microsoft Azure.
* **Рекомендация:** Инициализировать экземпляр `ConfidentialClientApplication` один раз как singleton внутри конструктора сервиса `__init__`, чтобы MSAL мог полноценно кэшировать токены в оперативной памяти бэкенда.

### 3. Отсутствие rate-limiting на логин по учетным записям (Brute-force)
* **Проблема:** Защита от перебора паролей на `/login` реализована на уровне Nginx (`infra/gateway/nginx.conf`) с ограничением 10 запросов в минуту по IP-адресу. Однако данное ограничение легко обходится злоумышленниками с помощью распределенных прокси-сетей (residental proxy). Механизм rate-limit по конкретному логину (`username`/`email`) на уровне приложения отсутствует.
* **Рекомендация:** Внедрить application-level rate limiting на эндпоинты авторизации по связке `username/email`, используя хранилище Redis или библиотеку `slowapi`.

## 🟠 P2 — Средний приоритет (Архитектурные и инфраструктурные риски)

### 4. Жестко закодированный часовой пояс (GMT+5)
* **Проблема:** Часовой пояс Азии/Алматы (`Asia/Almaty`, UTC+5) жестко зашит в коде отправки уведомлений Telegram-бота (`bot_service.py` и `telegram.py`). Это лишает систему гибкости при масштабировании на другие регионы и приведет к рассинхрону времени отображения переработок у сотрудников, находящихся в других часовых поясах.
* **Рекомендация:** Перенести конфигурацию часового пояса в настройки окружения (`Settings.DEFAULT_TIMEZONE` в `config.py`) или считывать его из профиля конкретного пользователя (`User.timezone`).

### 5. Риски компрометации в скрипте инициализации админа (`create_admin.py`)
* **Проблема:** Скрипт `backend/scripts/create_admin.py` создает пользователя `admin@example.com` с жестко зашитым в коде паролем `"admin"` и выставляет флаг `must_change_password=False`. Если данный скрипт будет по ошибке запущен администратором в продакшн-окружении, злоумышленники смогут получить полный контроль над системой.
* **Рекомендация:**
  1. Генерировать случайный пароль с помощью встроенного модуля `secrets`.
  2. Выводить сгенерированный пароль в stdout один раз при успешном создании.
  3. Всегда устанавливать флаг `must_change_password=True`.

### 6. Неявное сравнение дат refresh-токенов (tz-naive vs tz-aware)
* **Проблема:** Сравнение срока действия refresh-токена в `backend/app/services/refresh_token.py` опирается на то, что база данных вернет объект `datetime` с таймзоной (aware). В случае изменения драйвера БД или при тестировании на СУБД с другими типами колонок (например, SQLite), СУБД может вернуть datetime без таймзоны (naive), что вызовет фатальную ошибку `TypeError` в рантайме.
* **Рекомендация:** Явно нормализовать объект даты, проверяя наличие `tzinfo` перед осуществлением сравнения с текущим UTC-временем.

## 🟢 P3 — Низкий приоритет (Технический долг и оптимизация)

### 7. Зависимость от внешних CDN-сетей для UI
* **Проблема:** Страницы входа (`LoginPage.tsx`) и смены пароля (`ChangePasswordPage.tsx`) загружают фоновые изображения напрямую с серверов `images.unsplash.com`. При отсутствии интернета на клиенте, медленном соединении или блокировках провайдера, фон страниц авторизации не загрузится.
* **Рекомендация:** Скачать фоновые изображения в директорию `frontend/public/` и подключать их как локальные статические файлы.

### 8. Использование `text()` в методах сортировки ORM
* **Проблема:** В файле `backend/app/repositories/analytics.py` сортировка результатов по количеству часов переработки выполняется через конструкцию `order_by(text("total_hours DESC"))`. Это обходит типизацию SQLAlchemy и усложняет переносимость кода между диалектами БД.
* **Рекомендация:** Заменить сырой текст на стандартное выражение сортировки SQLAlchemy: `order_by(desc(total_hours))`.
